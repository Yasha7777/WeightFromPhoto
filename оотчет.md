## 2. Компоненты системы

### 2.1 Фронтенд (React, Vite)

**Расположение:** `/home/volumetric/frontend`

**Ключевые страницы:**
- `src/pages/Analyze.jsx` — загрузка фото, запуск анализа, polling результата, 3D-просмотр
- `src/components/PlyViewer.jsx` — интерактивный 3D-просмотрщик облака точек (Three.js / React Three Fiber)

**Поток данных на странице анализа:**
1. Пользователь выбирает фото → браузер читает EXIF через `exifr`
2. Фото сжимаются до 1600px через Canvas (QUALITY=0.85)
3. POST на `/api/analyses` с FormData (фото + EXIF как JSON)
4. Polling каждые 5 сек: `GET /api/analyses/:id`
5. При `status=completed` → показывает результат
6. При наличии `ply_url` → показывает `<PlyViewer />`

**PlyViewer:**
- Загружает PLY через `PLYLoader` из three.js
- После загрузки вычисляет bounding box → авто-позиционирует камеру
- Отображает облако точек с реальными цветами из фото (`vertexColors`)
- Ось Y инвертирована (`rotation={[Math.PI, 0, 0]}`) — компенсирует Y-down систему DUSt3R

### 2.2 CLIP-сервис

**URL:** `http://clip-service:8000`  
**Назначение:** векторизация изображений для поиска похожих куч в БД  
**Эндпоинт:** `POST /embed` → `{ embedding: float[], dim: int }`  
**Деплой:** docker-compose контейнер рядом с n8n

### 2.3 LLaVA (Ollama)

**URL:** `http://192.168.32.245:11434`  
**Модель:** `llava`  
**Назначение:** визуальная оценка объёма кучи по фотографии  

**Промпт включает:**
- Метаданные фото (сезон, GPS-направление, FOV из focal length)
- Подсказки из БД (похожие кучи с объёмами)
- Инструкции по геометрическому расчёту (конус, полуэллипсоид, клин)
- Систему скоринга уверенности (0.10–0.95)

**Выходной JSON:**
```json
{
  "material": "gravel",
  "material_type": "crushed",
  "L_m": 8.5, "W_m": 6.0, "H_m": 2.1,
  "shape": "half_ellipsoid",
  "ai_volume_m3": 56.2,
  "confidence": 0.72
}
```

## 4. GPU-сервер DUSt3R

### Алгоритм работы

```
n8n POST /run
  { analysis_id, photo_ids[], exif[] }
        │
        ▼
receiver_dust3r.py
  1. Сохранить exif.json в папку проекта
  2. Скачать фото из Supabase по photo_ids
  3. Запустить dust3r_gpu.py (в отдельном потоке)
  4. Прочитать _result.json
  5. Загрузить PLY в Supabase Storage
  6. PATCH analyses SET ply_url = ...
  7. Вернуть JSON в n8n
        │
        ▼
dust3r_gpu.py
  1. Загрузить модель из checkpoints/
  2. load_images(size=512) — ресайз до 512px по длинной стороне
  3. make_pairs(scene_graph='complete') — все пары фото
  4. inference() — нейросетевой инференс
  5. global_aligner(PointCloudOptimizer, niter=300)
  6. get_pts3d() + get_masks() + scene.imgs → облако точек
  7. Вычислить масштаб из GPS (EXIF)
  8. ConvexHull → объём
  9. Сохранить PLY (ASCII)
  10. Записать _result.json
```

### GPS-масштабирование

Масштаб вычисляется как соотношение расстояний между позициями камер:
- В метрах (GPS из EXIF, формула Haversine)
- В единицах DUSt3R (из `scene.get_im_poses()`)

Возвращаются два значения:
- `scale_weighted` — взвешенное среднее по топ-30% длинных пар GPS
- `scale_median` — медиана по всем парам

⚠️ **Известное ограничение:** GPS смартфона (точность 3-6м) при съёмке вплотную к куче (расстояния между камерами 1-5м) даёт погрешность масштаба ~60%, что при возведении в куб (объём = scale³) даёт погрешность объёма до 4×.

### Подсчёт объёма

Используется метод Convex Hull с зеркальным замыканием снизу:
```python
z_min = pts[:, 2].min()
mirrored = pts.copy()
mirrored[:, 2] = 2 * z_min - mirrored[:, 2]
hull = ConvexHull(vstack([pts, mirrored]))
volume = hull.volume / 2.0
```

### Ответ GPU-сервера

```json
{
  "status": "success",
  "analysis_id": "uuid",
  "point_count": 970633,
  "scale_weighted": 27.32,
  "scale_median": 25.00,
  "volume_dust3r_units": 0.005772,
  "volume_m3_weighted": 117.7,
  "volume_m3_median": 90.2,
  "ply_url": "https://supabase.gottland.ru/storage/v1/object/public/dust3r-ply/...",
  "ply_file": "./volmetric/projects/.../dust3r_output.ply"
}
```

## 5. n8n Workflow

**Инстанс:** self-hosted, версия 2.20.9

### Граф нод

```
Webhook (изображения + email)
        │
        ▼
EXIF → parse
        │
        ├──────────────────────────────────────────────┐
        │                                              │
        ▼                                              ▼
Prepare GPU Payload                              b64 → items
        │                                              │
        ▼                                              ▼
      dust3r ──────────────────┐        Эмбеддинг: CLIP
        │                      │                      │
        ▼                      │                      ▼
Сохранить PLY URL              │        БД: Поиск похожих куч
   (PATCH analyses)            │                      │
                               │                      ▼
                               │        Подготовка: Объединить ответы БД
                               │                      │
                               │                      ▼
                               │        Статистика: фильтр IQR + среднее + σ
                               │                      │
                               │                      ▼
                               │        Изображения → binary (для LLaVA)
                               │                      │
                               │                      ▼
                               │                Модель: LLaVA
                               │                      │
                               │                      ▼
                               │        Парсинг ответа LLaVA
                               │                      │
                               │                      ▼
                               │        IVW слияние + CI + σ_mass
                               │                      │
                               │                      ▼
                               └──────► Форматирование текста
                                               │
                                               ▼
                                       Собрать все фото в один item
                                               │
                                               ▼
                                           Сбор Email
                                               │
                                               ▼
                                       Loop Over Items
                                          │       │
                                          ▼       ▼
                                    HTTP-ответ  Почта
```

### Описание ключевых нод

#### `EXIF → parse`
Нормализует EXIF из браузера (exifr-формат) в единый формат:
```javascript
{ photo_index, exif: { lat, lon, focal_real, focal_35mm, orig_width, orig_height, ... } }
```

#### `Prepare GPU Payload`
Собирает payload для DUSt3R:
```javascript
{ analysis_id, photo_ids[], exif: exifParsed[] }
```

#### `dust3r`
HTTP POST на `http://143.55.45.86:42181/run`, timeout 25 мин.

#### `Сохранить PLY URL`
PATCH в `analyses` — записывает `ply_url` сразу после завершения DUSt3R, не дожидаясь LLaVA. Сайт получает 3D-модель по polling раньше финального результата.

#### `Эмбеддинг: CLIP`
Каждое фото → вектор через `clip-service:8000/embed` → pgvector-поиск в `heap_photos`.

#### `Статистика: фильтр IQR + среднее + σ`
Фильтрация результатов поиска:
1. Сортировка по similarity
2. Ratio-фильтр (MAX_RATIO=3.0 от топ-матча)
3. IQR-фильтр выбросов
4. Взвешенное среднее (вес = sim²)
5. Расчёт σ и CV

#### `IVW слияние + CI + σ_mass`
Байесовское слияние оценок AI и БД. Режимы:

| Режим | Условие | Логика |
|-------|---------|--------|
| `DB_ONLY_AI_OFF` | aiVolume ≤ 0.01 | Берём только БД |
| `EXACT_DB_MATCH` | sim ≥ 0.98 | 97% БД + 3% AI |
| `HIGH_DB` | sim ≥ 0.90 | Взвешенное IVW |
| `CONFLICT_SCALE_AI_PRIMARY` | ratio > 0.7, sim < 0.99 | AI первичен, БД с большим штрафом |
| `FUSED` | надёжная БД | Стандартное IVW |
| `AI_ONLY` | sim < 0.65 | Только AI |
| `NEW_MATERIAL` | sim < 0.4, conf > 0.75 | Новый материал |

#### `Форматирование текста`
Читает данные DUSt3R через `$('dust3r').first().json` (глобальный контекст выполнения), добавляет блок 3D-реконструкции к основному тексту результата.

### Docker (рекомендуется)

**Почему необходимо:** Vast.ai — аренда по часам. При потере инстанса теряется всё окружение. Docker-образ позволяет развернуть сервер на новом инстансе за 5 минут.

**Сборка образа:**
```bash
# На сервере
cd ~/dust3r
docker build -t volumetric-dust3r .
docker tag volumetric-dust3r your-dockerhub/volumetric-dust3r:latest
docker push your-dockerhub/volumetric-dust3r:latest
```

**Запуск на новом инстансе:**
```bash
# Создать папки для данных
mkdir -p /data/checkpoints /data/projects

# Скачать веса (один раз)
wget [https://download.europe.naverlabs.com/ComputerVision/DUSt3R/DUSt3R_ViTLarge_BaseDecoder_512_dpt.pth](https://download.europe.naverlabs.com/ComputerVision/DUSt3R/DUSt3R_ViTLarge_BaseDecoder_512_dpt.pth) \
     -P /data/checkpoints/

# Запустить контейнер
docker run -d \
  --gpus all \
  -p 6006:6006 \
  -v /data/checkpoints:/app/checkpoints \
  -v /data/projects:/app/volmetric/projects \
  --name dust3r \
  your-dockerhub/volumetric-dust3r:latest

# Логи
docker logs -f dust3r
```

**Что хранится вне образа (volumes):**
- `/data/checkpoints/` — веса модели (~3 ГБ, скачиваются один раз)
- `/data/projects/` — временные файлы анализов (можно не сохранять, PLY уходит в Supabase)

**Что хранится в образе:**
- Python-окружение и все зависимости
- Библиотека DUSt3R
- `receiver_dust3r.py` и `dust3r_gpu.py`

## 8. Известные ограничения

### GPS-масштаб объёма

GPS смартфона имеет точность 3-6 метров. При съёмке кучи с расстояния 1-3 метра погрешность масштаба составляет ~60%, что даёт погрешность объёма до 4× (ошибка возводится в куб).

**Текущий статус:** возвращаются оба варианта (`volume_m3_median`, `volume_m3_weighted`), объём помечается как ориентировочный.

**Решение:** маркер известного размера в кадре (кирпич, доска, паллета 1200×800 мм). Один предмет на одном фото даёт точность ±2-5%.
