## 1. Общая архитектура

```
Пользователь (браузер)
        │  загружает фото + EXIF
        ▼
   Фронтенд (React)
   gottland.ru
        │  POST /api/analyses  (FormData: фото, EXIF, title)
        ▼
   Supabase (PostgreSQL + Storage)
   supabase.gottland.ru
        │  webhook → n8n при создании analysis
        ▼
   n8n (оркестратор)
   Две параллельные ветки:
        │
        ├──► GPU-сервер DUSt3R (Vast.ai)        ─► PLY + GLB в Supabase Storage
        │    IP:порт меняется (см. раздел 5)         └► ply_url / glb_url → analyses
        │    масштаб берётся из калибровочного куба
        │    объём = интеграл высоты (heightfield)
        │
        └──► LLaVA + CLIP pipeline
             ├── CLIP (clip-service:8000)        ─► векторный поиск похожих куч в БД
             └── LLaVA (192.168.32.245:11434)    ─► определение материала и плотности
                      │
                      ▼
              IVW-слияние результатов (объём DUSt3R × плотность LLaVA = масса)
                      │
                      ▼
              Supabase: analyses.status = "completed"
                      │
                      ▼
              Email пользователю
```

> **Разделение ролей (актуально):**
> - **DUSt3R** даёт **объём** (3D-реконструкция + масштаб по кубу).
> - **LLaVA** даёт **материал и плотность** (объём LLaVA больше не используется — поле `ai_volume_m3` всегда `0`).
> - **CLIP + БД** дают справочную оценку по похожим кучам (эталоны).
> - Итоговая масса = объём DUSt3R × плотность LLaVA.

## 2. Компоненты системы

### 2.1 Фронтенд (React, Vite)

**Расположение:** `/home/volumetric/frontend`

**Ключевые страницы:**
- `src/pages/Analyze.jsx` — загрузка фото, запуск анализа, polling результата, 3D-просмотр
- 3D-просмотрщик — двойной формат: GLB-меш и PLY-облако точек (Three.js / React Three Fiber)

**Поток данных на странице анализа:**
1. Пользователь выбирает фото → браузер читает EXIF через `exifr`
2. Фото сжимаются до 1600px через Canvas (QUALITY=0.85)
3. POST на `/api/analyses` с FormData (фото + EXIF как JSON)
4. Polling каждые 5 сек: `GET /api/analyses/:id`
5. При `status=completed` → показывает результат
6. При наличии `glb_url` / `ply_url` → показывает 3D-просмотрщик

### 2.2 CLIP-сервис

**URL:** `http://clip-service:8000`
**Назначение:** векторизация изображений для поиска похожих куч в БД
**Эндпоинт:** `POST /embed` → `{ embedding: float[], dim: int }`
**Деплой:** docker-compose контейнер рядом с n8n

### 2.3 LLaVA (Ollama)

**URL:** `http://192.168.32.245:11434`
**Модель:** `llava`
**Назначение:** определение **материала** кучи и его **плотности** (кг/м³) по фотографии

**Промпт включает:**
- Метаданные фото (сезон, GPS-направление, FOV из focal length, качество по ISO/яркости)
- Подсказки из БД (похожие кучи с объёмами)
- Систему скоринга уверенности (0.10–0.95)

**Выходной JSON:**
```json
{
  "primary_subject": "pile of gravel",
  "material": "gravel",
  "material_type": "crushed",
  "density_kg_m3": 1500,
  "confidence": 0.72,
  "db_agreement": "agree"
}
```

> Объём LLaVA **не** оценивает — он приходит от DUSt3R.

### 2.4 GPU-сервер DUSt3R

**URL:** динамический (Vast.ai, см. [раздел 5](#5-где-менять-ip-адрес-gpu-сервера)).
На момент написания (22.06.2026): `http://106.246.35.133:40514/run`.
**ОС:** Ubuntu 24, Python 3.10, CUDA
**Рабочая директория:** `~/dust3r`

**Файлы:**

| Файл | Назначение |

## 4. GPU-сервер DUSt3R

### Алгоритм работы

```
n8n POST /run
  { analysis_id, photo_ids[], exif[] }
        │
        ▼
receiver_dust3r.py
  1. Сохранить exif.json в папку проекта
  2. Скачать фото из Supabase по photo_ids
  3. Запустить dust3r_gpu.py (в отдельном потоке)
  4. Прочитать _result.json
  5. Загрузить PLY и GLB в Supabase Storage (бакет dust3r-ply)
  6. Записать ply_url / glb_url / метрики в dust3r_results
  7. Вернуть JSON в n8n
        │
        ▼
dust3r_gpu.py
  1. Загрузить модель из checkpoints/
  2. load_images(size=512) — ресайз до 512px по длинной стороне
  3. Зафиксировать focal length из EXIF (если есть) — focal не «угадывается»
  4. make_pairs(scene_graph='complete') — все пары фото
  5. inference() — нейросетевой инференс
  6. global_aligner(PointCloudOptimizer, niter=300)
  7. get_pts3d() + get_masks() + scene.imgs → облако точек
  8. Найти масштаб по калибровочному кубу (compute_scale_from_cube)
  9. Изолировать целевой объект (RANSAC пол + DBSCAN, куб исключается)
 10. Объём = интеграл высоты над опорной плоскостью (heightfield)
 11. Сохранить PLY (ASCII) + GLB (визуальный меш)
 12. Записать _result.json
```

### Масштабирование по калибровочному кубу

Масштаб сцены (метры на единицу DUSt3R) определяется по шахматному паттерну на грани **калибровочного куба** известного размера, лежащего в кадре рядом с объектом:

1. `cv2.findChessboardCorners` ищет паттерн на каждом фото (полный кадр + 7 кропов + половинное разрешение).
2. Углы уточняются субпиксельно (`cornerSubPix`).
3. 2D-углы проецируются в 3D-облако DUSt3R, считаются расстояния между соседними углами.
4. Масштаб = `square_size_m / расстояние_в_единицах` (медиана по валидным парам, затем медиана по всем фото, где куб найден).
5. Sanity-check: масштаб должен попадать в правдоподобный диапазон, иначе считается ложным детектом и отбрасывается.

> Параметры куба (`square_size_m`, `pattern`) настраиваются в коде — см. [раздел 6](#6-где-менять-параметры-калибровочного-куба).
> Требования к фотосъёмке для надёжного детекта — см. [раздел 7](#7-инструкция-по-фотосъёмке-с-калибровочным-кубом).

### Изоляция целевого объекта

Перед подсчётом объёма из сцены вычленяется именно целевая куча:

1. **RANSAC** находит опорную плоскость (пол/асфальт) и срезает всё ниже неё, плюс тонкий слой сверху (`grass_trim`) — чтобы «оторвать» куб и траву от объекта.
2. **DBSCAN** кластеризует оставшиеся точки.
3. Кластеры рядом с найденным 3D-центром куба **исключаются** (`cube_exclusion_radius` ≈ 10–15 см) — куб не должен попасть в объём.
4. Целевой объект = **самый крупный** оставшийся (не-куб) кластер; вокруг него строится bbox с запасом 5%.

### Подсчёт объёма (heightfield / cut-and-fill)

Объём считается как **интеграл высоты над опорной плоскостью** (DEM): плоскость разбивается на сетку ячеек, в каждой ячейке берётся медианная высота точек, объём = Σ (высота ячейки × площадь ячейки). Метод устойчив к шуму у земли и подходит для куч произвольной формы.

ConvexHull с зеркальным замыканием снизу остался только как **fallback** на случай, если опорная плоскость не определена.

> Для калибровочных тестов в коде есть диагностика `diagnose_sphere_fit` — фитит сферу МНК и сравнивает радиус/объём полусферы с эталоном (например, 33 л / r≈25 см).

## 5. Где менять IP-адрес GPU-сервера

> **В коде менять НИЧЕГО не нужно.** `receiver_dust3r.py` поднимается на `host="0.0.0.0", port=6006` и не знает своего внешнего адреса. Адрес, по которому к серверу обращаются, хранится **в n8n**.

GPU-сервер арендуется на Vast.ai по часам, поэтому при пересоздании инстанса внешний IP и проброшенный порт **меняются**. Чтобы переключить систему на новый инстанс:

1. Открыть воркфлоу в **n8n**.
2. Найти ноду **`dust3r`** (HTTP Request, в верхней части графа, в блоке «Сервисы GPU-сервера»).
3. На странице арендованного сервера (Vast.ai) найти **внешний IP** и **порт, проброшенный на 6006**.
4. Вставить их в поле **URL** ноды `dust3r` в формате:

   ```
   http://<IP_сервера>:<внешний_порт_для_6006>/run
   ```

   Пример (текущий): `http://106.246.35.133:40514/run`

5. Сохранить воркфлоу. Перезапуск n8n не требуется.

> Нода `COLMAP` (альтернативный путь реконструкции) настраивается аналогично, но в текущей конфигурации отключена и не используется.

## 6. Где менять параметры калибровочного куба

Все параметры куба находятся в **`dust3r_gpu.py`**. Их два:

| Параметр | Что это | Значение по умолчанию |
|----------|---------|-----------------------|
| `square_size_m` | Сторона **одной клетки** шахматного паттерна на грани куба, **в метрах** | `0.01775` (= 7.1 см / 4 клетки) |
| `pattern` | Число **внутренних углов** паттерна `(cols, rows)` | `(3, 3)` |

**Как `pattern` зависит от размера шахматки:**
- грань 4×4 клетки → 3×3 = **9 внутренних углов** → `pattern=(3, 3)`
- грань 5×5 клеток → 4×4 = **16 внутренних углов** → `pattern=(4, 4)`

> Внутренние углы — это точки **пересечения** чёрных и белых клеток внутри паттерна (без крайних границ). Их всегда на единицу меньше числа клеток по каждой стороне.

### Где именно править

Оба параметра задаются в сигнатуре функции `compute_scale_from_cube` (≈ строка 36), но **реально применяются значения, которые передаются при вызове в `main()`** — они перекрывают дефолты сигнатуры. Поэтому менять нужно именно «рабочие» места:

**1. `square_size_m`** — управляется аргументом `--cube_square_m`, а так как `receiver_dust3r.py` его не передаёт, действует `default` в `argparse`:

```python
# dust3r_gpu.py → def main(), блок argparse
parser.add_argument("--cube_square_m", type=float, default=0.01775,
                    help="Размер одной клетки куба в метрах (7.1см / 4 = 0.01775)")
#                                       ^^^^^^^ менять здесь
```

**2. `pattern`** — задаётся прямо в вызове внутри `main()`:

```python
# dust3r_gpu.py → def main(), вызов функции
scale_cube, cube_3d_centers = compute_scale_from_cube(
    img_files, pts3d_np, masks_np,
    square_size_m=args.cube_square_m,
    pattern=(3, 3),          # ← менять здесь при другом размере шахматки
    dust3r_size=512
)
```

> ⚠️ Менять дефолт **только** в сигнатуре функции (строка 36) недостаточно: `main()` передаёт оба значения явно и перекрывает их. Правьте `argparse` (для `square_size_m`) и вызов в `main()` (для `pattern`).

## 7. Инструкция по фотосъёмке с калибровочным кубом

Цель — чтобы DUSt3R **одновременно** надёжно: (а) распознал шахматный паттерн на кубе (для масштаба) и (б) реконструировал **весь объект целиком** (для объёма). Это два разных требования, и фотографировать нужно с учётом обоих.

### Подготовка сцены

1. **Положите куб на ту же поверхность, что и объект** (тот же пол/асфальт). Масштаб переносится с куба на всю сцену, поэтому они должны стоять на одной плоскости.
2. **Отступ от объекта 15–30 см.** Куб не должен касаться кучи или быть «в ней»: алгоритм отделяет куб от объекта кластеризацией, и при слишком близком расположении куб может «слипнуться» с объектом или, наоборот, прихватить часть объекта в зону исключения.
3. **Шахматная грань куба обращена к камере** и хорошо видна. Грань должна быть чистой, не бликующей, без грязи и заломов.
4. **Фон рядом с кубом — однотонный.** Избегайте клетчатых/решётчатых/плиточных поверхностей в кадре: они дают ложные детекты шахматки.

### Съёмка

5. **Снимайте по кругу вокруг объекта**, 15–30 кадров (минимум 8–12). Чем больше перекрытие между соседними кадрами (≈ 40–60%), тем точнее реконструкция.
6. **Объект целиком в кадре** на большинстве снимков. Делайте кадры с разной высоты — на уровне глаз и чуть сверху, чтобы захватить **верх кучи** (без верха объём по высоте будет занижен).
7. **Куб должен быть чётко виден на НЕСКОЛЬКИХ кадрах** (не на одном). Масштаб берётся как медиана по всем фото, где куб найден, поэтому несколько детектов = устойчивее результат.
8. **Сделайте 2–4 кадра ближе к кубу**, где шахматка занимает заметную часть кадра и хорошо читается, плюс общие планы всего объекта. Сочетание «крупные планы куба + общие планы объекта» закрывает оба требования.
9. **Резкость и фокус.** Куб должен быть в фокусе, без смаза — субпиксельное уточнение углов требует чёткой картинки. Не двигайте телефон во время спуска.

### Освещение и качество

10. **Равномерный, рассеянный свет.** Избегайте прямого жёсткого солнца: блики и глубокие тени «убивают» детект углов шахматки и портят реконструкцию.
11. **По возможности низкое ISO** (пайплайн штрафует ISO > 1600). Без пере- и недодержки.
12. **EXIF не удалять.** Focal length из EXIF используется, чтобы зафиксировать фокусное расстояние камеры. Отправляйте оригиналы; некоторые мессенджеры вырезают метаданные — так фото слать нельзя.

### Чего избегать

13. Не меняйте сцену между кадрами (не двигайте куб и объект).
14. Не снимайте мокрые, зеркальные или прозрачные поверхности объекта — DUSt3R на них ошибается.
15. Не обрезайте объект так, чтобы он **ни на одном** кадре не был виден целиком — общими планами нужно покрыть его полностью.

### Описание ключевых нод

#### `EXIF → parse`
Нормализует EXIF из браузера (exifr-формат) в единый формат: GPS, фокусное (`focal_real`, `focal_35mm`), размеры оригинала, ISO/яркость, устройство, время съёмки.

#### `Prepare GPU Payload`
Собирает payload для DUSt3R: `{ analysis_id, photo_ids[], exif[] }`.

#### `dust3r`
HTTP POST на URL GPU-сервера (см. [раздел 5](#5-где-менять-ip-адрес-gpu-сервера)), timeout 25 мин.

#### `Сохранить PLY/GLB URL`
PATCH в `analyses` — записывает `ply_url` и `glb_url` сразу после DUSt3R, не дожидаясь LLaVA. Сайт получает 3D-модель по polling раньше финального результата.

#### `Эмбеддинг: CLIP` → `БД: Поиск похожих куч`
Каждое фото → вектор через `clip-service:8000/embed` → pgvector-поиск в `heap_photos` (RPC `match_heap_photos`).

#### `Статистика: фильтр IQR + среднее + σ`
1. Сортировка по similarity
2. Ratio-фильтр (MAX_RATIO=3.0 от топ-матча)
3. IQR-фильтр выбросов
4. Взвешенное среднее (вес = sim²)
5. Расчёт σ и CV, флаг `isDbReliable`

#### `LLaVA: только материал`
Отправляет топ-2 фото (по качеству EXIF) в LLaVA. Возвращает материал, тип, плотность (кг/м³), уверенность. **Объём не оценивает.**

#### `IVW слияние + CI + σ_mass`
Байесовское слияние оценок AI и БД для справочного объёма и массы. Режимы:

| Режим | Условие |
|-------|---------|
| `DB_ONLY_AI_OFF` | aiVolume ≤ 0.01 |
| `EXACT_DB_MATCH` | sim ≥ 0.98 |
| `HIGH_DB` | sim ≥ 0.90 |
| `CONFLICT_SCALE_AI_PRIMARY` | ratio > 0.7, sim < 0.99 |
| `FUSED` | надёжная БД |
| `AI_PRIMARY_WEAK_DB` / `AI_ONLY` | sim < 0.90 / < 0.65 |
| `NEW_MATERIAL` | sim < 0.4, conf > 0.75 |

#### `Форматирование текста`
Читает данные DUSt3R через `$('dust3r').first().json` (предпочитает `volume_m3`, затем медианный/взвешенный fallback), берёт плотность от LLaVA и считает итоговую массу = **объём DUSt3R × плотность LLaVA**. Формирует текст письма с блоком 3D-реконструкции и ссылкой на GLB-модель.

## 11. Известные ограничения

### Точность объёма зависит от детекта куба
Масштаб (а значит и объём) определяется по калибровочному кубу. Если куб не найден ни на одном фото, масштаб не вычисляется и объём в м³ не выдаётся (только в единицах DUSt3R). Поэтому критично соблюдать [инструкцию по съёмке](#7-инструкция-по-фотосъёмке-с-калибровочным-кубом).

Так как объём пропорционален масштабу в **кубе** (`объём ∝ scale³`), ошибка масштаба усиливается втрое. Надёжный детект куба на нескольких кадрах (медиана) снижает разброс.

### Куб мал для крупных объектов
Шахматка ~7 см рассчитана на дистанцию 0.3–2 м. Для больших куч издалека нужна более крупная мишень либо обязательные близкие кадры куба (см. раздел 7).

### Ложные детекты шахматки
Клетчатый/плиточный фон в кадре может быть распознан как калибровочный паттерн. В коде есть sanity-check по диапазону масштаба, но лучшая защита — однотонный фон рядом с кубом.

### Сложные поверхности
Мокрые, зеркальные, прозрачные и однородные без текстуры поверхности плохо реконструируются DUSt3R.

### Пилотная валидация
На калибровочной полусфере 33 л получена систематическая погрешность порядка −7…−15%. Результат пилотный, на реальных кучах требуется дополнительная валидация.
